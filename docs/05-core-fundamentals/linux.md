# Linux

Linux is an open-source Unix-like operating system kernel that powers 90%+ of web servers, cloud platforms, embedded systems, and supercomputers. Essential for backend, DevOps, and infrastructure roles.

## Key Concepts

| Concept | Description |
|---------|-------------|
| File Permissions | rwx for user/group/others; special: SUID, SGID, sticky bit |
| Hard vs Soft Link | Hard: shares inode, can't cross filesystem. Soft: points to path, can cross FS |
| Signals | SIGTERM (graceful), SIGKILL (force), SIGHUP (reload/hangup) |
| Process States | R (running), S (sleeping), D (disk sleep), Z (zombie), T (stopped) |
| Pipeline | Pipe stdout of one command to stdin of another: `cmd1 | cmd2` |
| Shell Scripting | Bash automation with variables, conditionals, loops, functions |
| Systemd | Modern init system with parallel startup, dependency-based activation |

## File System Hierarchy

```
/        Root
├── bin  Essential binaries
├── etc  Configuration files
├── home User directories
├── var  Variable data (logs)
├── tmp  Temporary files

├── dev  Device files

```

## Permission Numbers

| Number | Permission | Numeric |
|--------|-----------|---------|
| r (read) | 4 | Owner + Group + Others |
| w (write) | 2 | |

| Example: 755 | rwxr-xr-x | Owner: 7, Group: 5, Others: 5 |
| Example: 644 | rw-r--r-- | Owner: 6, Group: 4, Others: 4 |

## FAQ (Top 10)

**Q1: What is the difference between hard and soft links?**
Hard link shares inode with original; can't cross filesystems; survives deletion. Soft link has own inode; points to path; breaks if original deleted.

**Q2: What is the sticky bit?**
Permission bit on directories preventing users from deleting files they don't own. Used on `/tmp`. Set with `chmod +t dir`.

**Q3: What happens when you delete a file that's still open?**
Directory entry removed immediately, but data blocks freed only when last process closes the file descriptor. Disk usage doesn't decrease until then.

**Q4: What is the difference between SIGTERM and SIGKILL?**
SIGTERM (15): requests graceful termination, process can clean up. SIGKILL (9): forces immediate termination, cannot be caught or blocked.

**Q5: What is a zombie process?**
Terminated process whose parent hasn't called wait() to read its exit status. Occupies PID but uses no resources. Fix: parent calls wait() or kill parent.

**Q6: What does `set -euo pipefail` do?**
`-e`: exit on error. `-u`: error on unset variables. `-o pipefail`: pipeline returns exit status of last failed command.

**Q7: How do you find which process uses a port?**
`lsof -i :PORT` or `ss -tlnp | grep :PORT` or `fuser PORT/tcp`.

**Q8: What is the difference between `>` and `>>`?**
`>` overwrites the file. `>>` appends to the file.

**Q9: How do you set up a cron job?**
`crontab -e` with format: `minute hour day month weekday command`. Example: `0 2 * * * /path/script.sh` runs daily at 2 AM.

**Q10: What is /proc?**
Virtual filesystem containing process and system information. Not stored on disk; generated by kernel. Used by tools like `ps`, `top`, `free`.

## Text Processing

```bash
grep -r "pattern" /path      # Recursive search
sed 's/old/new/g' file       # Replace all occurrences
awk '{print $1, $3}' file    # Print columns 1 and 3
find / -type f -size +100M   # Find files > 100MB
```

## Common Mistakes

| Mistake | Solution |
|---------|----------|
| Using `rm -rf /` | Always double-check paths |
| Not quoting variables | Use `"$variable"` to prevent word splitting |
| Ignoring `set -e` | Start scripts with `set -euo pipefail` |
| Using `ls` in scripts | Use `find` or glob patterns |
| Running as root unnecessarily | Use sudo for specific commands only |

## Best Practices

1. Use full paths in scripts (`/usr/bin/grep` not `grep`)
2. Quote all variables to prevent word splitting
3. Use `set -euo pipefail` at script start
4. Log everything with tee or redirects
5. Use functions to avoid duplication
6. Validate inputs before using them
7. Test scripts with `set -n` (syntax check)

## Quick Reference

